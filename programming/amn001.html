Propositional Logic as Language

Lightning Quick Review

My long term goal: implement an automated theorem prover/checker, which uses Bourbaki's logic as foundations. This is hard, so I'll tackle it in many small steps.

We'll review Bourbaki's logical foundations, then consider their propositional logic as a formal language. Later articles will "grow" our consideration to include predicate logic (very hard), set theory (ug) and others (wimper).

Bourbaki's full logic consists of various operators: τ , ◻ , ∨ , ¬ , = , ∈ ⊃ Most of these are familiar, "=" for set equality, "∈" set membership, "∨" for the "or" logical operator, and "¬" (logical negation) are all routine...the others are foreign and frighten modern readers.

Bourbaki's peculiar notation establishes "⊃" to construct ordered pairs. We would write (a, b), Bourbaki would write a⊃b. Po-tay-to, Po-tah-to.

The symbols τ and ◻ are...uh, "quantifiers" no one has heard of. We won't worry about it now, but think of them as a pair which construct quantifiers.

How do we implement "and"? Simple:

a ∧ b := ¬ ( ( ¬ a ) ∨ ( ¬ b ) )

Implication is likewise trivial:

x ⇒ y := ¬ x ∨ y

TL;DR. We just need to implement "or" and "not" operators acting on boolean quantities.

Grammar for Propositional Logic

So what's a formal grammar for this propositional logic? Lets construct the formal grammar for propositional logic only, we won't worry about ordered pairs or the mysterious quantifiers yet. 

<boolean> ::= true | false
<expr> ::= <boolean>
        | ( <expr> )
        | not <expr>
        | <expr-1> or <expr-2>
<expr-1> and <expr-2> ::= not ((not <expr-1>) or (not <expr-2>))
<expr-1> implies <expr-2> ::= (not <expr-1>) or <expr-2>
<e-1> iff <e-2> ::= (<e-1> implies <e-2>) and (<e-2> implies <e-1>)

I tried to emphasize which words are "reserved" by making them bold, and introduced parentheses to keep track of the order of evaluation (i.e., to avoid ambiguities).

This is trivial to implement, so I won't worry too much about the details. The efficiency of the implementation, however, remains far from clear...and remains an open research field today(!).

Remark. We're not done yet! We need to specify how or and not transform boolean values. Once we specify this, we'll complete our construction of our formal system. As this is completely trivial, we leave it as an exercise to the reader. (End of Remark)
